# 개발자 포털 — 정보 구조 설계

커머스 플랫폼이 광고 API를 외부 개발자에게 처음 공개하는 상황에서 진행한 Ads API 개발자 포털 정보 구조 설계 케이스 스터디입니다.

[English →](/README.md)

<br>

## 배경

커머스 플랫폼은 순수 광고 플랫폼이 갖지 못한 데이터를 보유합니다. 자사 생태계 안에서 발생한 실제 거래 데이터와 행동 데이터입니다. 커머스 플랫폼이 광고 API를 외부 개발자에게 공개할 때, 이 데이터가 핵심 차별화 요소가 됩니다. 단, 개발자 포털이 이를 구조 안에서 드러낼 수 있어야 합니다.

이 케이스 스터디는 커머스 플랫폼의 첫 외부 공개 광고 API 개발자 포털 IA(Information Architecture) 설계를 다룹니다. 대행사·광고·측정 파트너를 대상으로 광고 인프라를 처음으로 외부에 공개하는 상황이었습니다.

설계 과제는 포털이 성장함에 따라 깔끔하게 확장되는 IA를 구축하면서, 플랫폼의 데이터 강점이 처음 설계 구조 안에서 보이도록 하는 것이었습니다.

<br>

## 설계 판단

### 기존 개발자 포털 구조를 따르지 않은 이유

기존 개발자 문서는 플랫폼 생태계에 이미 익숙한 내부 개발자를 전제로 설계되어 있었습니다. 객체와 스키마 설명이 먼저 나오고, 코드 블록이 좁고 부차적으로 배치된 구조였습니다. 도메인을 이미 아는 개발자에게는 적합합니다.

외부 광고 개발자의 접근 방식은 다릅니다. 이들은 구체적인 연동 목표를 갖고 포털에 도착하며, 실행 가능한 코드를 통해 빠르게 이해를 검증해야 합니다. 실행 예제에 도달하기 전에 객체 설명을 스크롤해야한다면 최초 연동 성공까지의 시간이 늘어납니다.

### 이 독자에게 더 맞는 구조

Payabli, Stripe, Eventbrite 개발자 포털을 분석한 결과, API 플레이그라운드 형식(Payabli)이 이 독자에게 가장 적합하다고 판단했습니다. 파라미터 확인·실행·응답 확인이 단일 뷰포트에서 이루어집니다.

| 포털 | 채택한 강점 | 이유 |
|---|---|---|
| Payabli | API 플레이그라운드 레이아웃 | 단일 뷰포트: 문서 + 파라미터 + 실시간 실행 |
| Stripe | 토글형 객체 패턴 | 중첩 객체를 기본 접기 처리해 인지 부담 감소 |
| Eventbrite | 섹션 기반 IA | 튜토리얼·하우투·레퍼런스·설명의 명확한 분리 |

**코드 우선, 단일 뷰포트 완결 레이아웃** — "파라미터를 읽는 단계"와 "파라미터를 테스트하는 단계" 사이의 전환을 없애기 위한 레이아웃입니다.

## 정보 구조

사이트맵은 두 가지 원칙을 기반으로 설계했습니다. 외부 개발자의 탐색부터 프로덕션 적용까지의 여정, 그리고 새로운 API와 SDK가 추가될 때의 확장성입니다.

```
Ads API 개발자 포털
│
├── 시작하기                     # 인증 · 빠른 시작 · 연동 흐름 개요
├── 개념 가이드                  # 광고 객체 위계 · 캠페인 유형
│                                # 전환 추적 · 기여 모델 · 광고 정책
├── 연동 가이드                  # 계정 설정 · 광고 운영
│                                # 성과 관리 · 전환 추적
├── API Reference                # 계정 · 캠페인 · 성과 리포트
│                                # 전환 이벤트
├── SDK                          # JavaScript/Node.js · Python · Java · PHP · Ruby
├── 확장 영역                    # 잠재고객 · 플랫폼 고유 데이터 기반 연동
│                                # 크리에이티브 · 최적화 · 실험 관리
├── 공통 레퍼런스                # 에러 코드 · 메트릭 정의
│                                # Rate Limiting · 페이지네이션
└── 변경 이력 (Changelog)        # RSS 구독 · Breaking Change 공지
```

**핵심 구조 결정:**

**에러 코드와 메트릭 정의는 공통 레퍼런스에 통합합니다.**

엔드포인트마다 정의를 반복하면 정의가 변경될 때 모든 페이지를 수정해야 합니다. 공유 참조 자료의 단일 출처는 어떤 규모에서도 필수입니다.

**개념 가이드는 연동 가이드보다 먼저 위치합니다.** 

기여 모델, 전환 윈도우, 광고 객체 위계 같은 광고 플랫폼 개념이 낯선 외부 개발자는 이 기반 없이 연동 가이드를 완료할 수 없습니다. 개념 문서를 생략하면 숙련된 개발자의 진입 시간은 줄어들지만, 나머지 모든 개발자의 서포트 부담이 늘어납니다.

**Changelog는 최상위 섹션입니다.**

API 변경은 주석이 아닙니다. 이미 연동한 외부 개발자는 API가 변경될 때 신뢰할 수 있는 구독 가능한 신호가 필요합니다. RSS 지원과 Breaking Change 별도 공지는 나중에 추가하는 것이 아니라 첫날부터 갖춰야 합니다.

**확장 영역은 설계 단계부터 확장 가능하도록 구성합니다.**

플랫폼 고유 데이터 연동, 크리에이티브 도구, 최적화 기능을 한 섹션에 모아두면 포털이 성장해도 핵심 연동 경로(시작하기 → 연동 가이드 → API Reference)가 깔끔하게 유지됩니다.

## 기능 우선순위

| 순위 | 기능 | 우선하는 이유 |
|---|---|---|
| 1 | **API 플레이그라운드** | 최초 연동 성공까지의 시간을 줄입니다. 포털을 떠나거나 Postman을 설정하지 않고도 파라미터 확인 → 실행 → 응답 확인이 가능합니다. |
| 2 | **플랫폼 고유 데이터 기반 연동** | 플랫폼이 보유한 고유 사용자 데이터는 외부 광고 플랫폼이 제공할 수 없는 타겟팅을 가능하게 합니다. 포털의 핵심 차별화 요소이므로 나중에 추가하는 것이 아니라 IA 설계 단계부터 포함해야 합니다. |
| 3 | **토글형 객체 패턴** | 중첩 객체를 기본 접기 처리합니다. API 표면이 늘어날수록 인지 부담이 관리 가능한 수준을 유지합니다. Stripe의 패턴을 채택했으며, 코드를 가리지 않고 복잡한 스키마를 처리합니다. |
| 4 | **실험 관리** | 캠페인 A/B 실험을 진행하는 대행사는 구조화된 측정이 필요합니다. Meta와 Google 모두 이를 최상위 섹션으로 운영합니다. |

## 문서 업데이트 프로세스

`openapi.yaml`을 API 스펙의 단일 정본으로 관리합니다. 스펙 파일이 업데이트되면 문서가 자동으로 동기화되어 문서가 낡는 문제를 구조적으로 예방합니다.

목표는 문서를 최신 상태로 유지하는 것이 아닙니다. 스펙이 변경되는 순간부터 외부 개발자가 성공적으로 적응하는 순간까지, 세 주체가 동일선상에 있도록 연결하는 것입니다.

```
엔지니어 (기술적 의도)  →  TW (영향 판단 · 번역)  →  외부 개발자 (연동 목표)
```

**프로세스:**

```
스펙 변경
  → TW: PR 리뷰 · Breaking Change 여부 판단 · 마이그레이션 가이드 작성
  → 내부 선공유 (고객팀 · 서포트)
  → 외부 공지 (Changelog · 이메일 · Breaking Change 별도 안내)
  → 배포
  → 모니터링: 에러 로그 · 서포트 티켓 · 문서 이탈률
  → 실패 신호 감지 시 문서 개선 → 품질 기준 업데이트
```

**TW가 머지 후 스펙이 아닌 PR 단계에서 리뷰하는 이유:**

PR은 의도가 보이는 단계입니다. 머지 후에는 diff로 무엇이 바뀌었는지는 알 수 있지만, 왜 바뀌었는지, 그리고 이미 연동한 개발자에게 무엇을 의미하는지는 알 수 없습니다. PR 단계에 TW가 개입하면 그 맥락이 커밋 히스토리 속으로 사라지기 전에 포착할 수 있습니다.

**문서는 배포 시점에 완성되지 않습니다.** 

배포 후 외부 개발자의 반응이 문서가 놓친 부분을 알려주는 신호입니다. 모니터링 단계는 선택이 아닙니다. 이 단계가 외부 개발자의 반응을 다시 문서 개선으로 연결하는 순환을 만듭니다.
