# 정보 구조 설계: 개발자 포털 케이스 스터디

<br>
<br>

## 이 문서에서 얻는 것

이 문서를 다 읽으면, 하나의 개발자 포털 구조 뒤에 있는 전체 판단 과정을 알게 됩니다. 왜 긴 설명 대신 코드 우선 실행 방식으로 만들었는지, 실제 개발자 포털 다섯 곳을 분석해서 얻은 결정이 무엇인지, 그리고 API가 늘어나도 사이트맵이 깔끔하게 유지되는 방법까지요.

<br>
<br>

## 배경

물류 플랫폼은 대부분의 경쟁사가 갖지 못한 데이터를 보유하고 있습니다. 자사 네트워크 안에서 실시간으로 생성되는 배송 신호입니다. 지금 이 순간 화물이 어디 있는지, 배송 시간이 어떻게 변하고 있는지, 어느 경로가 성과가 낮은지 같은 데이터입니다. 이 플랫폼이 이 데이터를 담은 API를 외부 개발자에게 처음 공개할 때, 이 데이터가 포털의 핵심 강점이 됩니다. 단, 개발자가 이를 쉽게 찾고 쓸 수 있어야만 그렇습니다.

이 케이스 스터디는 이 플랫폼의 첫 외부 공개 개발자 포털 정보 구조(IA) 설계를 다룹니다. 소매업체, 풀필먼트 파트너, 연동 벤더가 처음으로 이 추적 API에 연동하는 상황을 대상으로 합니다.

설계 과제는 이랬습니다. API가 계속 추가되어도 깔끔하게 확장되는 IA를 만들면서, 플랫폼의 데이터 강점이 나중에 덧붙이는 게 아니라 처음 설계 구조부터 보이게 하는 것입니다.

<br>
<br>

## 설계 판단

<br>

### 실제로 이 포털을 읽는 사람은 누구인가

플랫폼의 기존 내부 문서는 회사 안에서 이미 시스템을 잘 아는 엔지니어를 대상으로 만들어져 있었습니다. 객체와 데이터 모델에 대한 긴 설명이 먼저 나오고, 코드 예제는 좁고 부차적으로 배치돼 있었습니다. 도메인을 이미 아는 사람에게는 문제없는 구조입니다.

외부 개발자는 다르게 접근합니다. "이 API를 내 시스템에 연동한다"는 하나의 구체적인 목표를 갖고 도착하며, 실제 코드를 최대한 빨리 실행해서 방향이 맞는지 확인하고 싶어합니다. 실행 가능한 예제에 도달하기 전에 객체 설명 화면을 몇 번이나 넘겨야 한다면, 그만큼 첫 성공까지의 시간이 늘어납니다.

<br>

### 왜 하필 이 레이아웃인가

실제 개발자 포털 다섯 곳, Payabli, Stripe, Twilio, Plaid, Eventbrite를 분석한 결과, 이 독자에게 가장 잘 맞는 패턴 하나가 뚜렷하게 드러났습니다. 문서, 파라미터, 실시간 실행이 한 화면 안에 함께 있는 API 플레이그라운드 레이아웃입니다.

| 분석한 포털 | 채택한 요소 | 이유 |
|---|---|---|
| Payabli | API 플레이그라운드 레이아웃 | 한 화면에 문서 + 파라미터 + 실시간 실행이 함께 있음 |
| Stripe | 접기형 객체 패턴 | 중첩된 데이터 객체가 기본으로 접혀 있어, API가 커져도 화면이 복잡해지지 않음 |
| Eventbrite | 섹션 기반 구조 | "배우기", "문제 해결하기", "찾아보기", "이해하기"의 명확한 분리 |

결정: **코드 우선, 단일 화면 레이아웃**입니다. "이 파라미터가 뭘 하는지 읽는 단계"와 "직접 실행해서 결과를 보는 단계" 사이의 전환을 없애기 때문입니다.

## 사이트 구조

사이트맵은 두 가지 원칙을 따릅니다. 외부 개발자가 "방금 발견함"에서 "프로덕션에서 운영 중"까지 실제로 이동하는 순서와 일치해야 하고, 나중에 새로운 API가 추가돼도 쉽게 확장할 수 있어야 합니다.

```
개발자 포털
│
├── 시작하기                  # 인증 · 빠른 시작 · 연동 흐름 한눈에 보기
├── 개념 가이드                # 핵심 데이터 모델 · 배송 이벤트 작동 방식
│                             # 배송 시간대 · 추적 정확도 · 데이터 보관 정책
├── 연동 가이드                # 계정 설정 · 추적 요청 보내기
│                             # 배송 이벤트 처리하기 · 에러 복구
├── API Reference              # 배송(Shipments) · 배송 이벤트 · 웹훅
├── SDK                        # JavaScript/Node.js · Python · Java · PHP · Ruby
├── 확장 영역                  # 과거 데이터 조회 · 맞춤 이벤트 필터
│                             # 대량 처리 · 샌드박스 테스트
├── 공통 레퍼런스              # 에러 코드 · 필드 정의
│                             # Rate Limit · 페이지네이션
└── 변경 이력 (Changelog)      # RSS 피드 · Breaking Change 공지
```

**가장 중요한 구조 결정들:**

- **에러 코드와 필드 정의는 공통 레퍼런스 하나에만 있고, API 페이지마다 반복하지 않습니다.** 같은 정의를 10개 페이지에 복사해두면, 그 정의가 바뀔 때마다 10개 페이지를 전부 고쳐야 합니다. 하나의 공유 출처만 있으면 어떤 규모에서도 이 문제 자체가 발생하지 않습니다.

- **개념 가이드가 연동 가이드보다 먼저 있습니다.** 배송 이벤트, 배송 시간대, 추적 정확도가 어떻게 작동하는지 모르는 개발자는 이 기반 없이 현실적으로 연동 가이드를 끝낼 수 없습니다. 연동 가이드로 바로 건너뛰면 이미 도메인을 아는 개발자는 더 빨리 진행할 수 있지만, 나머지 모두는 막혀서 서포트를 필요로 하게 됩니다.

- **Changelog는 확장 영역에 묻혀 있지 않고 최상위 섹션입니다.** 이미 연동을 마친 개발자에게 API 변경 사항은 중요합니다. 믿고 확인할 수 있는 곳, 이상적으로는 구독까지 가능한 곳이 필요합니다. RSS 지원과 별도의 Breaking Change 공지는 출시 시점부터 있어야지, 첫 사고가 나고 나서 추가하면 늦습니다.

- **확장 영역은 처음부터 확장을 염두에 두고 설계했습니다.** 과거 데이터 조회, 대량 처리, 샌드박스 도구를 한곳에 모아둬서, 고급 기능이 더 추가돼도 핵심 경로인 시작하기 → 연동 가이드 → API Reference는 계속 깔끔하게 유지됩니다.

## 무엇을 먼저 만들었고, 왜인가

| 순위 | 기능 | 이유 |
|---|---|---|
| 1 | **API 플레이그라운드** | 개발자의 첫 성공까지의 시간을 줄입니다. 포털을 벗어나거나 별도의 테스트 도구를 설정하지 않고도 파라미터를 확인하고, 실행하고, 응답을 볼 수 있습니다. |
| 2 | **문서 안에 실시간 배송 데이터를 직접 노출** | 이 API를 일반적인 배송 추적 API 대신 쓸 이유를 만드는 건 플랫폼의 실제 운영 데이터입니다. 나중에 덧붙이는 게 아니라 처음부터 구조 안에 있어야 합니다. |
| 3 | **접기형 객체 패턴** | 중첩 객체가 기본으로 접혀 있어서, 필드가 늘어나도 페이지가 읽기 어려워지지 않습니다. 실제 코드를 가리지 않으면서 복잡한 데이터를 다루는 Stripe의 방식을 채택했습니다. |
| 4 | **대량 처리 지원** | 수백 건의 배송을 한 번에 추적하는 풀필먼트 파트너에게는 이게 부가 기능이 아니라 기본 기능이어야 합니다. 단일 배송 엔드포인트를 반복 호출하며 억지로 처리하게 만들면 안 됩니다. |

**나중으로 미룬 것과 그 이유:** 검색과 API 버전 관리는 고려했지만 초기 출시에는 넣지 않았습니다. 검색은 포털이 약 15페이지를 넘어설 때부터 실제로 필요해집니다. 그전에는 사이드바만 훑어봐도 충분합니다. 버전 관리는 실제로 Breaking Change가 발생했을 때부터 의미가 있습니다. 둘 다 이미 만든 구조를 다시 짜지 않고도 나중에 추가할 수 있습니다.

## 문서를 실제 API와 계속 일치시키기

API 스펙 파일이 단일 정본입니다. 스펙이 바뀌면 문서가 자동으로 업데이트되어, "문서에는 이렇게 나와 있는데 실제 API는 다르게 동작한다"는 종류의 버그 자체가 사라집니다.

하지만 문서를 정확하게 유지하는 것과 개발자를 성공시키는 것은 다른 문제입니다. 진짜 목표는 스펙이 바뀌는 순간부터 영향받는 개발자가 자신의 연동을 실제로 조정하는 순간까지, 모두가 같은 정보를 보고 있게 만드는 것입니다.

```
엔지니어 (무엇이, 왜 바뀌었는지 앎)  →  Technical Writer (영향을 판단하고, 이를 번역함)  →  외부 개발자 (무엇을 해야 하는지 알아야 함)
```

**프로세스:**

```
스펙 변경
→ Technical Writer가 변경 내용을 검토 · Breaking Change 여부 판단 · 필요시 마이그레이션 가이드 작성
→ 외부 개발자보다 먼저 내부 팀(서포트, 파트너 성공팀)에 공유
→ 외부 공지 발송 (Changelog · 이메일 · 해당되면 별도 Breaking Change 알림)
→ 변경 사항 배포
→ 모니터링: 에러 로그 · 서포트 티켓 · 개발자가 성공하지 못하고 문서를 떠나는 비율
→ 명백히 작동하지 않는 부분이 있으면 → 문서 수정 → 품질 체크리스트 업데이트
```

**Technical Writer가 머지된 최종 결과가 아니라 변경 요청 단계에서 리뷰하는 이유.** 변경 요청 단계는 그 변경의 의도, 즉 무엇이 바뀌었는지가 아니라 왜 바뀌었는지가 아직 보이는 단계입니다. 머지되고 나면 diff로 무엇이 달라졌는지는 알 수 있지만, 왜 바뀌었는지, 그리고 이미 예전 버전으로 연동을 마친 개발자에게 이게 무엇을 의미하는지는 알 수 없습니다. 더 이른 단계에서 리뷰하면 이 맥락이 커밋 히스토리 속으로 사라지기 전에 붙잡을 수 있습니다.

**문서는 발행된 순간 끝나는 게 아닙니다.** 배포 후 외부 개발자가 실제로 어떻게 반응하는지가 문서가 무엇을 놓쳤는지 알려주는 진짜 신호입니다. 모니터링은 선택이 아닙니다. 개발자가 실제로 겪는 경험과 다음에 고쳐야 할 것 사이의 순환을 완성하는 단계입니다.

## 이 설계를 위해 분석한 포털

| 포털 | 분석한 내용 |
|---|---|
| Payabli | API 플레이그라운드 레이아웃 · 단일 화면 실행 |
| Stripe | 접기형 중첩 객체 패턴 |
| Twilio | 도메인이 낯선 개발자를 위한 개념 가이드 우선 구조 |
| Plaid | 샌드박스 환경을 실제 문서와 나란히 제공하는 방식 |
| Eventbrite | 개발자 목표 기준의 섹션 구조 |
