# 작동 방식: 선택부터 스트리밍 출력까지

## 이 문서에서 얻는 것

[아키텍처](../ko/architecture-ko.md) 문서는 HiDoc이 규제 지식을 4개 레이어로 구조화하는 방법을 설명합니다. 이 문서는 실제로 사용자가 시스템을 실행할 때 무슨 일이 일어나는지를 설명합니다. "GDPR을 선택하고 회사 프로파일을 입력했다"에서 "완성된 문서가 화면에 섹션별로 나타난다"까지 이어지는 6단계 흐름입니다.

이 문서를 다 읽으면, 각 단계가 정확히 무엇을 하는지, 정보가 없을 때 엔진이 왜 추측하지 않고 멈추는지, 그리고 스트리밍 방식이 어떻게 작동하는지 알게 됩니다.

<br>

## 6단계 흐름, 한눈에 보기

```
사용자가 규제를 선택하고 회사 프로파일을 입력
              ↓
엔진이 atom_output_mapping을 조회
↓
엔진이 회사 프로파일 조건을 적용
↓
엔진이 injection_order에 따라 원자를 정렬
    ↓
출력 섹션별 프롬프트 조립
↓
Claude API가 섹션 내용을 생성
↓
SSE로 클라이언트에 스트리밍
```

<br>

## 1단계 — 사용자 입력

생성이 시작되기 전, 사용자는 두 가지를 입력합니다.

**규제 선택:** 준수하고자 하는 규제를 하나 이상 선택합니다. 예를 들어 GDPR과 ISMS-P를 함께 선택할 수 있습니다. 규제를 선택하면, [아키텍처](../ko/architecture-ko.md) 문서에서 설명한 원자, 즉 더는 쪼갤 수 없는 하나의 규제 의무 중 그 규제에 매핑된 모든 원자가 활성화됩니다.

**회사 프로파일:** 회사의 제품이 실제로 무엇을 하는지, 데이터를 어떻게 다루는지를 설명하는 구조화된 필드 집합입니다. 이 프로파일이 바로 어떤 **조건부 원자**, 즉 `condition`이 `NULL`이 아닌 원자들이 이 특정 회사에 실제로 적용되는지를 결정합니다.

**생성 결과에 영향을 주는 프로파일 필드:**

| 필드 | 효과 |
|---|---|
| `processes_special_category_data` | `if_special_category_data` 조건의 원자를 활성화 |
| `uses_consent_as_legal_basis` | `if_uses_consent` 조건의 원자를 활성화 |
| `has_data_processor_agreements` | 데이터 처리 계약 관련 출력물을 활성화 |
| `operates_in_eu` | GDPR의 국경 간 데이터 이전 관련 원자를 활성화 |
| `company_name`, `dpo_contact` | 조건이 아니라, 생성된 텍스트에 실제 값으로 곧바로 삽입되는 필드 |

<br>

## 2단계 — 원자 조회

엔진은 1단계에서 사용자가 선택한 규제에 속하는 모든 원자-출력물 매핑을 조회합니다.

```sql
SELECT aom.*, ag.source_clause, ag.principle_context
FROM atom_output_mapping aom
JOIN atom_generation ag ON aom.atom_id = ag.id
WHERE ag.regulation_id IN (selected_regulation_ids)
ORDER BY aom.output_id, aom.injection_order
```

쉽게 말하면, 사용자가 선택한 규제와 관련된 모든 원자를 가져오되, 이미 어떤 출력물에 속하는지 기준으로 묶여 있고, 그 출력물 안에서 등장해야 할 순서대로 이미 정렬된 상태로 가져옵니다.

<br>

## 3단계 — 조건 필터링

모든 원자가 모든 회사에 적용되는 건 아닙니다. 바로 이때 [아키텍처](../ko/architecture-ko.md) 문서에서 소개한 `condition` 컬럼이 쓰입니다. 이 단계에서는 각 원자의 조건을 1단계에서 입력한 회사 프로파일과 대조합니다.

```
atom.condition = NULL                       → 항상 주입
atom.condition = 'if_uses_consent'          → profile.uses_consent = true일 때만 주입
atom.condition = 'if_special_category_data' → profile.processes_special_category_data = true일 때만 주입
```

조건이 맞지 않는 원자는 생성이 일어나기도 전에 여기서 제외됩니다. 이 과정 덕분에 최종 문서가 가능한 모든 의무를 언급하는 일반적인 템플릿처럼 읽히지 않고, 실제로 이 회사가 어떻게 운영되는지에 맞춰집니다.

<br>

## 4단계 — 섹션 조립

엔진이 생성해야 하는 각 출력물에 대해, 그 출력물에 기여하는 모든 원자를 모아 `injection_order` 순서대로 배치합니다. [아키텍처](../ko/architecture-ko.md) 문서의 레이어 3에서 설명한 것과 같은 순서 개념입니다.

조립된 각 섹션은 생성 모델에 전달되는 하나의 프롬프트가 됩니다.

```
System: [생성 엔진 제어 원칙]
[스타일·톤 지침]
[출력 유형 컨텍스트: DOC_ / EVD_ / OPR_ / TCH_ / CTR_ / POL_]
User:   [원자 1 원문 조항 + principle_context]
[원자 2 원문 조항 + principle_context]
...
[회사 프로파일 값]
[출력 목표: DOC_PP_RETENTION 생성]
```

<br>

## 5단계 — 생성 엔진 제어

모든 생성 호출은 예외 없이 네 가지 원칙을 따릅니다. 이 원칙들은 HiDoc이라는 시스템 전체가 원래 피하려고 설계된 것, 즉 실제 규제 근거 없이 내용을 지어내는 것을 모델이 하지 못하게 막기 위해 존재합니다.

**Information Discipline**
엔진은 원자 매핑에서 명시적으로 허가한 내용만 주입합니다. 어떤 부분에 대한 원자 내용이 없으면 그 부분은 아예 생성되지 않습니다. 모델이 그럴듯해 보이는 추측으로 빈칸을 채울 여지 자체가 없습니다.

**Mapping Completeness**
어떤 출력물의 소스 원자가 완전히 매핑되어 있지 않으면, 그 섹션은 아예 생성되지 않습니다. 시스템은 최선을 다한 추측 대신 명시적인 경고를 내놓습니다. 자신 있게 들리지만 불완전한 컴플라이언스 문서가, 명백히 빠져 있는 문서보다 더 위험하기 때문입니다.

**Generation Traceability**
생성된 모든 섹션은 어떤 원자가 여기에 기여했는지 정확히 기록합니다. 덕분에 누구든 완성된 문서 속 한 문장을 그것이 나온 구체적인 규제 조항까지 거슬러 추적할 수 있습니다.

**`[INFORMATION NEEDED]` 프로토콜**
생성에 필요한 회사 프로파일 값이 없으면, 엔진은 추측하거나 조용히 생략하는 대신 문서에 `[INFORMATION NEEDED: field_name]`이라는 실제 텍스트 플레이스홀더를 삽입합니다. 문서는 구조적으로는 완성된 상태를 유지하고, 사용자는 그 빈칸 하나만 채우면 됩니다.

<br>

## 6단계 — 스트리밍 출력

각 섹션은 Claude API(모델: claude-sonnet-4-6)로 전달되고, 그 응답은 Server-Sent Events(SSE) 방식으로 클라이언트에 스트리밍됩니다. SSE란 응답이 완전히 준비될 때까지 기다렸다가 한 번에 보내는 게 아니라, 준비되는 대로 조각조각 브라우저에 전송하는 방식입니다.

섹션은 최종 문서에 나타나는 순서 그대로 스트리밍됩니다. 그래서 클라이언트는 각 섹션이 도착하는 즉시 화면에 표시할 수 있고, 사용자는 모든 게 끝날 때까지 로딩 화면을 보는 대신 문서가 실시간으로 만들어지는 과정을 지켜보게 됩니다.

<br>

## View 모드

같은 생성 문서라도 읽는 사람에 따라 세 가지 View 모드로 보여줄 수 있습니다.

| 모드 | 대상 | 보여주는 내용 |
|---|---|---|
| `PUBLIC_VIEW` | 최종 사용자, 고객 | 개인정보처리방침과 소비자 공개용 내용만 — 내부 정보는 없음 |
| `INTERNAL_REVIEW` | 법무·컴플라이언스 담당자 | 전체 문서에, 각 부분을 뒷받침하는 원문 조항 주석까지 포함 |
| `AUDIT_SUBMISSION` | 규제 기관, 감사인 | 의무 추적 태그가 명시적으로 포함된, 외부 검토용으로 정리된 문서 |

<br>

## 톤 설정

같은 내용이라도 대상 독자에 따라 네 가지 톤으로 생성할 수 있습니다. 의무 자체는 바뀌지 않고, 표현 방식만 달라집니다.

| 톤 | 대상 | 어떤 느낌인가 |
|---|---|---|
| `FORMAL_REGULATORY` | 규제 기관, 법무팀, 엔터프라이즈 독자 | 수동태, 조항 번호 인라인 표기, 법률 용어를 그대로 유지 |
| `PROFESSIONAL_ACCESSIBLE` ★ 기본값 | B2B SaaS, 스타트업 | 능동태, 법률 용어 최소화, 전문적인 비즈니스 문체 |
| `FRIENDLY_PLAIN` | 소비자 앱, B2C | "당신"에게 직접 말하는 방식, 짧은 문장 |
| `INDUSTRY_TECHNICAL` | 개발자 대상 B2B, API 제품 | 기술 용어, 구체적인 수치, 구현자 관점에서 서술 |
