# HiDoc — Hybrid Documentation

> AI가 컴플라이언스 문서 초안을 생성합니다. 사람이 검토하고, 수정하고, 승인합니다.

[English →](../README.md)

<br>

## 이 프로젝트가 해결하는 문제

같은 컴플라이언스 의무가 개인정보처리방침, 보안 통제, 계약서 조항처럼 여러 곳에 함께 등장해야 하는 경우가 많습니다. 그런데 대부분의 도구는 규제마다 완전히 별개의 문서를 생성합니다. 그러면 팀은 같은 요구사항의 여러 버전을 각각 관리해야 하고, 그 버전들이 서로 어긋나는 일도 흔합니다. 게다가 컴플라이언스 문서는 법적 정밀함을 요구하는데, 대부분의 팀에는 이런 전문성이 사내에 없습니다. 결국 위험할 정도로 일반적인 상용구를 쓰거나, 비용이 큰 외부 전문가에게 맡기게 됩니다.

HiDoc은 회사의 제품 프로파일과 선택한 규제를 입력받아, 모든 섹션이 그 섹션이 충족하는 규제 의무까지 추적 가능한 단일 통합 문서를 생성합니다. 한 섹션이 GDPR 제13조와 ISMS-P를 동시에 충족하면, 그 섹션은 한 번만 작성됩니다. 두 개의 별도 문서에 중복으로 작성되지 않습니다.

<br>

## 간단히 보는 작동 방식

HiDoc은 모든 규제를 **원자(atom)**, 즉 하나의 조항에서 추출한 더는 쪼갤 수 없는 최소 단위의 컴플라이언스 의무로 쪼갠 다음, 각 원자를 그것이 속하는 출력물, 즉 문서 섹션, 증거 자료, 기술적 통제, 계약, 정책, 운영 절차 중 해당하는 곳에 매핑합니다.

```
입력:  회사 프로파일 + 선택한 규제 (예: GDPR + ISMS-P)
엔진:  원자 기반 컴플라이언스 매핑
출력:  단일 통합 문서 — 각 섹션에 충족 규제 조항 태그 포함
```

→ 이 구조 뒤에 있는 전체 판단 과정: [아키텍처](../en/architecture.md) (한국어: [아키텍처](architecture-ko.md))
→ 입력부터 스트리밍 출력까지 단계별 생성 흐름: [작동 방식](../en/how-it-works.md) (한국어: [작동 방식](how-it-works-ko.md))

## 지금까지 만들어진 것

**Phase 1 MVP 완료:**

- GDPR Art.5–Art.33: 79개 원자 전체 매핑 완료, 한국 법정 보존 의무 5개 원자 추가
- `compliance_outputs`: 6개 출력 유형에 걸쳐 97개 출력 정의
- `atom_output_mapping`: 147개 원자-출력물 매핑
- SSE 스트리밍을 갖춘 생성 엔진 운영 중
- 개인정보처리방침(9개 섹션, 47개 원자): 처음부터 끝까지 전체 생성 검증 완료

**다음 단계:** EU AI Act 원자 구축 · DPA 생성 엔진 · Company Profile 입력 UI

## 내 역할

**창업자 — 제품 설계, 아키텍처, AI 보조 개발**

HiDoc은 의료기기 산업에서 30개국 이상의 컴플라이언스 문서화를 8년간 담당한 경험에서 시작했습니다. 원자 모델, 4레이어 아키텍처, 출력 유형 체계, 생성 엔진 제어 원칙은 모두 직접 설계했습니다. 저는 프로그래머가 아니라서, 구현 전 과정을 Claude를 개발 파트너로 활용해 AI 보조 방식으로 진행했습니다.

목표는 항상 필요했지만 존재하지 않았던 도구를 만드는 것이었습니다. 규제 구조를 충분히 깊이 이해해서, 처음부터 통합적이고 추적 가능하며 감사에 통과할 수 있게 설계된 문서를 생성하는 도구입니다.
